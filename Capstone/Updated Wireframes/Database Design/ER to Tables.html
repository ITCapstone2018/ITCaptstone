Introduction

Mapping an E-R Diagram to tables is a lot like an architect moving from whiteboard drawings to a blueprint. In essence, an E-R Diagram shows an idea of how given data relates to each other in terms of entities and relationships. Tables, on the other hand, take the ideas and concepts of data interaction outlined in the E-R Diagram, but make them database ready. Moving an E-R Diagram to a table consists of identifying primary keys, placing foreign keys, and defining cardinality.

Rules for Mapping E-R Diagrams and Tables

When mapping an ERD to a table, the first consideration is to ensure that every entity, or table, has a unique, identifiable value called the primary key. Ideally, the primary key is short, numeric, does not change, and can never be null. Therefore, surrogate keys, or automatically generated numerical values, make perfect candidates for a table's primary key. Relationships between tables are created through the use of foreign keys. A foreign key is the primary key of one table that is placed into another table to create a relationship. The foreign key can not just be placed into any table, however; there are rules that govern its placement based on its type of relationship.

Foreign Key Rules

1:1 Relationship 
Usually does not matter where the foreign key is placed. Should be placed in the most used entity. If an entity in the relationship is optional, the foreign key goes with the optional entity. 

1:N Relationship 
The One side is called the parent, the Many side is called the child. In this relationship, the foreign key always goes in the many side of the relationship. 

N:M Relationship 
In a N:M Relationship, there is no place for the foreign key in either table. N:M Relationships will always be mapped to two 1:N relationships. To do this, a third table, called an intersection table, is created that represents the relationship itself. The primary key of the intersection table is a composite key of both primary keys from the related entities.
